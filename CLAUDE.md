# AppTrainingPWA

PWA (Progressive Web App) qui reproduit à l'identique l'app de coaching **AppTraining officielle** (Google Apps Script). Hébergée sur GitHub Pages, backend partagé avec le GAS.

- **URL live** : https://yohangrsbrtn.github.io/AppTraining/
- **Backend** : `https://script.google.com/macros/s/AKfycbwQiM6ixf-CTIWwcuNHoosFbvrDzWmC056yRUGhTaWv0Nwxbm0dLeK3d5QVgqmS7P9G7A/exec` (Web App GAS, même code que l'app officielle — voir `../AppTraining/Code.js`)
- **Référence visuelle/fonctionnelle** : `../AppTraining/Index.html` et `../AppTraining/Code.js` sont la source de vérité. Objectif permanent : la PWA doit ressembler et se comporter *exactement* comme l'app GAS officielle, sauf demande explicite contraire.

## Déploiement

- Push sur `main` = déploiement automatique (GitHub Pages sert directement depuis le repo). Pousser après chaque modification, sans demander confirmation.
- Le GAS (`Code.js`/`Index.html`) est un projet **séparé**, non versionné sur GitHub, déployé via `clasp push && clasp deploy --deploymentId <id>` depuis `/Users/yohangrosbertin/Desktop/AppTraining`.

## Architecture

- `index.html` — coquille de l'app (CSS, état global `S`, routage `setPage()`/`render()`, accueil, classement, profil, réglages)
- **`console.html`** — tableau de bord desktop pour le coach (noir + or, sidebar/dashboard/tableau/panneau de détail, coins arrondis ~16px, bordures quasi invisibles — volontairement *pas* le style "app mobile" du reste de la PWA, cf. maquette validée). Fichier HTML autonome (charge juste `api.js`), pas de login dédié : lit `localStorage` (`at_coach`/`at_token`) déjà posé par une connexion coach faite dans `index.html`, sinon écran "Connecte-toi d'abord". Pages réelles v1 : **Accueil** (cartes Bilans à traiter/en retard, Activité récente, Activité 7 jours, Aliments à valider) et **Clients** (tableau tri/recherche/filtre + panneau latéral + actions protocole/verrouillage/vue client) — tout branché sur les endpoints coach existants (`listerClientsAvecNiveaux`, `listerBilansCoach`, `chargerTousLesLogs`, `listerAlimentsCommunauteAValider`, `verifierRetardBilan` en fan-out par client, `definirChimieActif`, `verrouilerAccesClient`/`deverrouilerAccesClient`), aucune nouvelle fonction backend. **Piège** : `listerClientsAvecNiveaux()` renvoie un `JSON.stringify(...)` côté serveur (même famille de piège que `validerBilan`) — toujours `JSON.parse` côté client, contrairement à `listerBilansCoach`/`chargerTousLesLogs`/`listerAlimentsCommunauteAValider` qui renvoient déjà de vrais objets. "Ouvrir en vue client" navigue vers `index.html?viewAs=<id>` ; un petit bootstrap au boot de `index.html` lit ce paramètre et appelle `enterVueClient()` automatiquement si le coach est connecté. **Pages Bilans/Classement/Base alimentaire/Protocole** (v2) réutilisent les données déjà chargées au boot (`data.bilansCoach`, `data.clients`, `data.aValider`) sans appel réseau supplémentaire, sauf Base alimentaire qui refetch après suppression (piège `deleteRow` déjà documenté ci-dessus). Mensurations/Paramètres restent en placeholder (pas de donnée bilan disponible en masse sans fan-out coûteux par client). **`data.clients` exclut systématiquement `yohan`** (le coach lui-même) — sinon il apparaît comme un client dans Classement/Protocole/tableau Clients. **Onglet Notes (panneau détail client)** : première brique hors Google Sheets — table `notes_coach` sur **Supabase** (Postgres), appelée en REST direct depuis `console.html` (`SUPABASE_URL`/`SUPABASE_ANON_KEY`, constantes en haut du script) via la clé `anon` publique + policy RLS ouverte à `anon` (même niveau de sécurité que le reste de l'app — pas de vrai système d'auth Supabase pour l'instant). CRUD simple (`chargerNotes`/`ajouterNote`/`supprimerNoteApi`), aucun impact sur la PWA client ni sur le backend GAS. Pensé comme point de départ d'une migration progressive ("strangler fig") : les nouvelles fonctionnalités coach sans équivalent Sheets partent directement sur Supabase, l'existant reste sur GAS tant qu'il n'y a pas de vraie raison de le migrer. **Page Programmes (bibliothèque d'exercices + templates)** : même principe Supabase que Notes — coach-only, aucun impact Sheets/client, construite directement en production. Cinq tables, RLS ouverte à `anon` comme `notes_coach` : `exercices`, `programme_templates`, `programme_blocs`, `programme_seances`, `programme_seance_exercices`. Deux sous-onglets : **Exercices** — nom, groupe musculaire (liste alignée sur la nomenclature réelle du user : Pectoraux/Dos (dorsaux)/Dos (trapèzes)/Dos (global)/Epaules (anterieur/latéral/posterieur)/Biceps/Triceps/Abdos/Lombaires/Quadriceps/Ischios/Fessiers/Adducteurs/Abducteurs/Mollets/Cardio), `groupe_secondaire` optionnel, équipement, lien vidéo YouTube/Instagram (simple URL, pas d'upload), notes, **et un jeu de reps/repos par bloc de périodisation** (`reps_metabolique`/`repos_metabolique`, `reps_mecanique`/`repos_mecanique`, `reps_force`/`repos_force`) — bibliothèque préremplie une fois (138 exercices) depuis le Google Sheet de référence du user (import ponctuel via script, pas un flux réutilisable) ; recherche + filtre par groupe. **Templates** — nom/catégorie/description, puis une arborescence **Bloc → Séance → Exercice** :
  - Un **bloc** = une phase de périodisation optionnelle (`type`: `metabolique`/`mecanique`/`force`/`null`="Générique", + `nombre_semaines`). Un nouveau template démarre avec un seul bloc générique invisible dans l'UI (pas de chrome bloc affiché tant qu'il n'y en a qu'un et qu'il reste générique) — le coach n'est jamais obligé de penser en blocs. Le bouton "+ Ajouter un bloc de périodisation" révèle les contrôles (type + nombre de semaines) sur tous les blocs dès qu'il y en a plus d'un.
  - Une **séance** = un jour titré librement (`titre` texte, ex: "Séance 1 — Push"), remplace l'ancien "Jour N" muet.
  - Un exercice dans une séance garde `series/reps/repos/tempo/notes` (comme avant). Le picker de dosage (`choisirExercicePourSeance`) propose de préremplir reps/repos depuis les valeurs par bloc de l'exercice — si la séance appartient à un bloc typé, ce bloc est présélectionné et préremplit automatiquement les champs ; les 2 autres blocs restent dispo en un clic pour un choix manuel.
  `exercice_id` référencé par `programme_seance_exercices` est `on delete restrict` : supprimer un exercice utilisé dans une séance renvoie une erreur Supabase (409, violation FK) affichée telle quelle côté coach plutôt que d'échouer silencieusement — retirer l'exercice de la séance d'abord. Sauvegarde d'un template = upsert de la ligne `programme_templates` puis suppression complète (cascade sur blocs→séances→exercices) + réinsertion de toute l'arborescence (même pattern que `modifierMenu` côté GAS pour les menus clients) — le mapping bloc→séance→exercice après réinsertion se fait par recherche sur la colonne `ordre` (jamais en supposant que Postgres renvoie les lignes d'un insert multi-lignes dans l'ordre d'origine, ce qui n'est pas garanti). Supprimer un template cascade automatiquement sur tout (`on delete cascade` en chaîne). v1 = bibliothèque seule, **pas d'assignation à un client réel** — c'est la prochaine étape prévue : elle écrira dans le programme réel d'un client et devra donc passer par le bac à sable (`AppTraining-Sandbox`/`AppTrainingDatabase`) avant d'être portée en production, contrairement à cette bibliothèque qui ne touche aucune donnée client. **Page Diètes (bibliothèque de templates)** : même principe que Programmes — coach-only, construite directement en production, v1 = bibliothèque seule sans assignation client. Trois tables Supabase (RLS `anon` identique) : `diete_templates`, `diete_repas` (repas titré librement, ex: "Petit-déjeuner"), `diete_repas_aliments`. Contrairement à Programmes (qui référence `exercices` par FK), un aliment dans un repas est un **snapshot** — nom/source/valeurs macro par gramme et quantité copiés au moment de l'ajout, jamais une référence live — parce que les aliments viennent de Google Sheets (`chargerBaseAliments()`, Code.js) et n'ont pas d'id stable exploitable : les entrées de la base coach (`copie de base`) n'ont aucun id du tout, et le `ligne` des entrées communautaires (`AlimentsCommunaute`) se décale à chaque suppression ailleurs dans la feuille (même piège `deleteRow` que documenté plus haut). Même pattern déjà utilisé par `MesMenus`/`JournalDiete` côté client. Les macros (par aliment/repas/jour) ne sont **jamais stockées** : recalculées à l'affichage (`recalcMacrosAffichage()`) depuis `quantite_g × valeur_par_gramme`, mis à jour en direct (sans re-render complet, juste `textContent` des cellules concernées) quand la quantité change, pour ne pas perdre le focus du champ en cours de frappe. `sucres_g`/`fibres_g`/`ags_g` valent `null` (inconnu) pour les aliments de la base coach — affichés "—", jamais confondus avec 0 ; sommés comme 0 dans les sous-totaux/totaux (même convention que `diete.js`). La recherche d'aliment (`chargerBaseAlimentsCache()`, appelée via `api()` sans `apiAs` — `chargerBaseAliments`/`ajouterAlimentCommunaute` ignorent le paramètre `client`) charge la base une fois en mémoire puis filtre/trie côté client (algorithme porté de `diete.js`), avec des chips Tous/Ma base/Communauté. Créer un nouvel aliment réutilise `ajouterAlimentCommunaute` telle quelle (déjà en production via `diete.js` et la page de modération Base alimentaire — aucun nouveau risque), avec en plus un champ code-barres optionnel + bouton "Chercher via Open Food Facts" (`world.openfoodfacts.org`, portage desktop sans caméra du flux de scan mobile de `diete.js`).
- `api.js` — `api()`/`apiAs()` vers le backend GAS. `_viewAsClientOverride` : quand posé, TOUTES les requêtes `api()` utilisent ce client au lieu du client connecté — c'est le mécanisme qui alimente la "vue client" du coach (`enterVueClient`/`exitVueClient` dans `index.html`).
- `bilan.js`, `training.js`, `diete.js`, `mensurations.js`, `recettes.js`, `progression.js`, `collection.js`, `coach.js`, `protocole.js` — une page par fichier
- **Préchargement** (`index.html`, section "Préchargement") : `_pf` (cache), `schedulerPrechargement()`/`precharger()` relancent le chargement en arrière-plan de Bilan/Programme/Diète/Mensurations à CHAQUE atterrissage sur l'accueil OU sur l'une de ces 4 pages elles-mêmes (pas juste une fois) — navigation quasi instantanée tant que le cache n'a pas été consommé. `_pfSession` s'incrémente à chaque bascule d'identité (login, `enterVueClient`/`exitVueClient`) pour ignorer les résultats d'un préchargement lancé pour un autre client.
- **`apiEtendreBilan()`** (`api.js`) : sérialise tous les appels à `chargerBilan`/`chargerJourneeEnCours` (voir piège ci-dessous) via une file d'attente commune — à utiliser systématiquement pour ces deux actions, y compris depuis le préchargement, `verifierEtCocherTraining`, etc., pas seulement depuis l'accueil.
- **XP/niveau dynamiques sur l'accueil** : `rafraichirProgressionEtDeblocages()` (`collection.js`) est le seul point de vérité à appeler après tout gain d'XP (séance/journée/bilan) — elle refetch `chargerProgressionClient`, patche `_pf.home.prog` (évite un cache périmé au prochain retour accueil), rafraîchit en place `carteHeaderHome` (`_majCarteHeader()`, `index.html`) si on est sur l'accueil, et déclenche `verifierDeblocages`. Ne jamais patcher `S.data.prog` à la main pour un déblocage (ancien bug dans `validerEtEnvoyerConfirme`/`bilan.js`, ex-`confirmerCloture` : ne mettait à jour que `niveau`, laissant XP/barre périmés) — toujours passer par cette fonction.
- **`showLoadingOverlay()`/`hideLoadingOverlay()`** (`index.html`) : popup léger (fond grisé + petite carte + spin + message, façon GAS) pour les actions/navigations courtes — à utiliser à la place de `setPage('loading')` (spinner plein écran) sauf pour le tout premier chargement après connexion.

## Pièges connus (ne pas refaire les mêmes erreurs)

- **`chargerBilan` et `chargerJourneeEnCours` appellent toutes les deux `etendreBilan()` côté serveur** (écriture sur la feuille Bilan). Ne JAMAIS les appeler en parallèle — ça plante. `loadHome()` utilise `chargerJourneeEnCours` (pas `chargerBilan`, jamais lue par l'accueil).
- **`validerBilan` renvoie une chaîne `JSON.stringify`** côté serveur (contrairement à `validerJournee`/`envoyerBilanAuCoach` qui renvoient de vrais objets) — toujours faire `typeof raw === 'string' ? JSON.parse(raw) : raw`.
- **Bouton bilan client fusionné** : un seul bouton "Envoyer au coach" (plus de "Clôturer" séparé) — `doEnvoyerBilanAuCoach` (`bilan.js`) vérifie le retard puis ouvre le récap existant (`ouvrirRecapBilan`), dont le bouton de confirmation appelle `validerEtEnvoyerConfirme` → action serveur unique `validerEtEnvoyerBilan(client, ligneTitre, targetSunday)` (Code.js) qui enchaîne `validerBilan()` (calcul XP) puis `envoyerBilanAuCoach()` (notif coach + bonus ponctualité) **sans réécrire leur logique** — réutilisées telles quelles pour ne pas retoucher aux anti-doublons XP déjà débogués (cf. pièges ci-dessus). Renvoie aussi un `JSON.stringify` (même piège). Si le bloc était déjà clôturé sous l'ancien système (`dejaValide=true` historique), l'erreur `bilan_deja_valide` de `validerBilan` est absorbée silencieusement — l'envoi se fait quand même, jamais bloquant. Les champs restent modifiables même après envoi (seule la ré-validation/ré-envoi est bloquée) — pas de recalcul XP après coup, comportement déjà présent avant la fusion.
- **Feuille Bilan, bloc de 9 lignes** : col C = date de validation (double usage : présence = bilan validé). Col U (21) = case "Valide Bilan" **fusionnée sur les 7 lignes de jours** (une seule valeur visuelle pour toute la semaine, écrite uniquement par `validerBilan`). Col V (22) = validation quotidienne par jour (`validerJournee`), non fusionnée, ajoutée spécifiquement pour ne pas entrer en conflit avec U.
- **Mensurations sont totalement indépendantes du bilan** — pas de blocage à l'envoi/clôture (le blocage historique a été retiré). Ne pas réintroduire de vérification bloquante.
- **Règle permanente : aucun champ de saisie (`input`/`textarea`/`select`) ne doit avoir `font-size` < 16px** — en dessous, Safari iOS zoome automatiquement la page au focus et ne dézoome pas correctement ensuite (bug constaté et corrigé une première fois sur `.bilan-input`/`.bilan-textarea`/`.t-select`/`.set-input` dans `index.html`, puis systématiquement appliqué à tout nouveau champ créé depuis, ex: tous les inputs de `diete.js`/Mes menus/Mon journal). À vérifier sur tout nouveau champ ajouté, dans n'importe quel fichier — que ce soit via une classe CSS partagée ou un style inline.
- **Le "Guide & Fonctionnement" dans Paramètres doit rester synchro avec le comportement réel** — ne pas copier bêtement le texte du GAS s'il décrit un ancien comportement (ex: l'ancien blocage mensurations).
- Le système de déblocage niveau/titre (confettis, glow) est dans `collection.js` (`verifierDeblocages`/`afficherDeblocage`) — déclenché après journée/séance/bilan/envoi-avec-bonus via `rafraichirProgressionEtDeblocages()`, et au chargement de l'accueil via `_apresChargementHome()` (`index.html`). Si un nouveau point de gain d'XP est ajouté, penser à y accrocher cette vérification. Ces deux points d'entrée n'excluent QUE `_viewAsClientOverride` (vue client) — pas `S.isCoach` : le compte coach voit ses propres passages de niveau/titres exactement comme un client (aligné sur le GAS officiel, où rien n'exclut le coach). Piège déjà rencontré : un `!S.isCoach` ajouté par erreur rendait ces popups invisibles pour le coach sur son propre profil.
- Palette de couleurs alignée sur le GAS (`index.html` `:root`) : fond page `#0f1117`, fond carte `#1a1d29`, bordure `#2d3142`, texte secondaire `#8892a4`. Les cartes GAS n'ont **pas** de bordure visible — uniquement le contraste de fond.
- `getBadgeSVG`/`TITRES_DEF`/`BADGES_NIVEAU` (badges de niveau + titres) sont dans `collection.js`, utilisés partout (accueil, progression, profil, collection, classement, coach). Toute modification de style s'y fait une seule fois et se propage automatiquement partout — vérifier tous les call sites avant de considérer un changement "fini".
- **`logActivite(client, programme, type, semaine, xp)` (Code.js) déduplique sur `client|programme|semaine|type`** — le paramètre `semaine` DOIT être unique par occurrence réelle (numéro de semaine, ou date), jamais une valeur qui se répète chaque semaine (ex: nom du jour "Mardi") — sinon `mettreAJourStats` (qui met à jour le total XP affiché) est silencieusement sauté à partir de la 2e occurrence, alors que la ligne `Log` est quand même ajoutée (bug réel trouvé et corrigé sur `validerJournee`/bonus pas — cause du total XP qui n'augmentait plus). **Même bug retrouvé et corrigé sur les bilans** (`logBilanAvecBonuses` et `envoyerBilanAuCoach`, Code.js) : `semaine` y était dérivé de `data[base][1]`, qui vaut littéralement le texte constant `"SEMAINE"` (l'en-tête du bloc détecté par `trouverBlocsBilan`) — identique pour TOUS les bilans de TOUS les clients, donc `mettreAJourStats` n'incrémentait plus jamais `Stats!XP`/`Stats!Bilans` après le tout premier bilan validé de chaque client, ce qui gelait à 0-1 le compteur de bilans et sous-comptait l'XP affichés sur la page Progression. Corrigé en utilisant `'L' + ligneTitre` (numéro de ligne du bloc, garanti unique — `etendreBilan` n'ajoute que du nouveau, ne réutilise jamais une ligne) comme clé de dédoublonnage.
- **Post-mortem du repair de `Stats!XP`/`Stats!Bilans`** (suite au point précédent) — deux pièges rencontrés en essayant de recalculer l'historique, à connaître avant toute nouvelle tentative de ce genre :
  1. **Les comptes clients ont été recréés récemment sur la PWA** (bascule depuis l'ancien système GAS, cf. [[project_migration_pwa]]) — `Clients!D` (date 1ère connexion, aussi utilisée comme `dateDebut` par `chargerProgressionClient` pour calculer `bilansAttendus`) date de cette bascule (juillet 2026 pour la plupart), alors que la feuille `Bilan` de chaque client est continue et contient tout l'historique validé sous l'ancien système. Un premier recalcul basé sur TOUS les blocs validés de la feuille Bilan (sans filtrer sur `Clients!D`) a donc compté des dizaines de bilans pré-migration comme si le client venait de les valider dans la PWA (ex: 27 pour un client, contre 2 réellement depuis son inscription sur la PWA). **Toute vérité terrain tirée de la feuille Bilan doit être restreinte aux blocs dont la date de validation est ≥ `Clients!D`.**
  2. **La date de validation d'un bloc Bilan et l'horodatage du Log ne sont pas comparables jour pour jour.** `validerBilan` date le bloc via `maintenant()` (décalage de 2h : avant 2h du matin = encore la veille) et/ou le dimanche de la semaine visée (`targetSunday`), alors que `logActivite` horodate le Log via `new Date()` brut, sans ce décalage — un bilan validé à 00h09 est donc écrit "la veille" dans la feuille Bilan mais "le lendemain" dans le Log. Comparer les jours calendaires entre les deux sources perd silencieusement ces bilans (XP retombée à ~0 pour l'un d'eux dans une tentative de recalcul). Pour retrouver l'XP d'un bilan/bonus_ponctualite historique, dédoublonner les lignes Log **entre elles** par fenêtre de temps (~15 min — un vrai doublon est un re-clic à quelques minutes d'écart, jamais à plusieurs jours), jamais en les comparant à la date du bloc Bilan.
  - Conclusion pratique : le compteur de **bilans/séances** (colonnes D/E de `Stats`) a été recalculé de façon fiable avec cette méthode (vérité terrain = feuille Bilan/Log restreints à `Clients!D`, dédoublonnage par fenêtre de temps) et corrigé en base pour tous les clients existants — pas besoin de le refaire. L'**XP** (colonne C), elle, est repassée en correction manuelle par le coach suite à plusieurs tentatives automatiques jugées peu fiables sur les comptes anciens (trop peu de lignes Log pour couvrir tout l'historique pré-migration) — ne pas retenter de recalcul automatique de l'XP sans en rediscuter avec lui d'abord.
- **Calcul "bilan en retard" (bonus ponctualité +20 XP à l'envoi, `envoyerBilanAuCoach`, et badge de retard, `verifierRetardBilan`)** : logique dupliquée dans ces 2 fonctions de Code.js, basée sur `joursDepuis = (jourNow - jourBilan + 7) % 7` où `jourBilan` vient de `Dashboard!D11`. `enRetard` doit être `joursDepuis >= 2 || (joursDepuis === 1 && minutesNow > 730)` — une version antérieure testait `joursDepuis === 2 || joursDepuis === 3` (sans `>=`), ce qui traitait à tort les jours 4/5/6 après la deadline comme "dans les temps" (bonus XP accordé plusieurs jours en retard). Si l'une des deux fonctions est retouchée, penser à répercuter le même correctif sur l'autre.
- **`chargerTousLesLogs` (Code.js) doit connaître TOUS les types de log existants** (`TYPES_CONNUS` : `seance`, `bilan`, `journee`, `bilanEnvoye`, `bonus_ponctualite`, `bonus_streak`, `bonus_pas`) pour distinguer le "nouveau format" (colonne Type) de l'"ancien format" (avant l'ajout de cette colonne, où la case contenait directement la semaine). Un type absent de cette liste est traité à tort comme ancien format : le type est écrasé en `'seance'` et la valeur du type se retrouve dans le champ `semaine`, ce qui donne des libellés absurdes façon "Séance validée · bonus_ponctualite" dans le centre de notifications coach. Tout nouveau type passé à `logActivite(...)` doit être ajouté à cette liste, ET son libellé d'affichage ajouté dans `renderNotificationsCoach` (`coach.js`, PWA) et `afficherLogs` (`Index.html`, GAS — même logique dupliquée, à tenir en synchro). La déduplication de `chargerTousLesLogs` inclut aussi le `type` dans sa clé (`client|programme|semaine|type`) — sans ça, deux événements distincts partageant le même `programme`/`semaine` (ex: "Bilan clôturé" et "Bilan envoyé au coach", tous deux `programme:'Bilan semaine'`) s'écrasent l'un l'autre.
- **Toute nouvelle action serveur qui accorde de l'XP doit fixer le montant côté serveur, jamais le recevoir du client** (`validerSeanceActivite` par ex. ignore un éventuel `xp` envoyé par le client) — sinon un client peut s'auto-attribuer de l'XP arbitraire via un appel direct à l'action.
- **Mode simplifié** (`localStorage['modeSimplifie']`, réglage par appareil, pas par client — comme le GAS) cache XP/progression/classement et coupe les popups de déblocage/XP ; toute nouvelle UI liée à l'XP doit vérifier `modeSimplifieActif()` si elle est censée être masquée dans ce mode.
- **Protocole "chimie"** (`protocole.js`) : lecture seule, ne jamais écrire dans la feuille `Protocol` du client — la saisie (dates, molécules) reste manuelle côté coach dans Sheets, l'app recalcule juste l'affichage à la volée. Activable individuellement par client via le toggle 🧬 dans "Mes clients" (coach), colonne J de la feuille `Clients`. Page à onglets (`_protocoleTab` : `cycle`/`analyses`).
- **Onglet "Analyses" du Protocole** (suivi prise de sang) : même principe lecture seule, sur une feuille `Analyses` du client (format tidy : colonnes `Date | Marqueur | Valeur | Unité | RefMin | RefMax`, une ligne par marqueur/date). Pas d'extraction PDF automatisée côté app — le coach passe les photos/PDF des résultats dans un chat dédié avec Claude qui ressort un xlsx prêt à coller dans cette feuille. `chargerAnalysesSante(client)` (Code.js) regroupe par marqueur, calcule le statut (bas/normal/haut) selon RefMin/RefMax, gated par le même flag `ChimieActif` que le reste du Protocole.
- **Notifications push (FCM)** : implémentées et déployées, mais cassées sur iPhone du coach (bug iOS non résoluble, diagnostiqué en profondeur) — mises de côté sur sa demande. Le code reste en place, ne pas re-déboguer sans nouvel élément concret.
- **Feuille `Clients` (LOG_SHEET_ID)** : col A=id, **B=nom, C=hash MotDePasse (SHA-256), D=date 1ère connexion**, F=niveau, **G=dernière connexion (affichage coach, pas le mot de passe !)**, H=titreActif, I=token push FCM, J=ChimieActif. Piège déjà rencontré : confondre colonne G (dernière connexion) et colonne C (mot de passe réel) en pensant réinitialiser un mot de passe.
- **Login PWA en 2 temps** (`index.html`, `_loginStep`: `identifiant`→`password`→`create`) : `verifierClient` puis toujours l'écran mot de passe classique (jamais de pré-aiguillage sur `clientAMotDePasse` avant saisie — ça court-circuiterait l'accès coach, dont le code maître `COACH_CODE` est comparé en premier côté serveur indépendamment de tout mot de passe stocké). Le passage à l'écran "Crée ton mot de passe" ne se déclenche QUE si le login échoue et que le serveur renvoie explicitement l'erreur `sans_mot_de_passe` (colonne C vide pour ce client) — jamais de manière prédictive. `creerMotDePasse` (Code.js, exposé via `doPost`) exige que le client existe déjà dans `CLIENTS` avant d'écrire quoi que ce soit, sinon n'importe quel appel HTTP non authentifié pourrait créer une ligne arbitraire dans la feuille Clients.
- **Base alimentaire communautaire** (`diete.js` + `coach.js`) : `chargerBaseAliments()` (Code.js) fusionne deux sources en lecture — la base maître du coach (fichier séparé `BASE_ALIMENTAIRE_ID`, onglet `copie de base`, jamais modifié par l'app) et un onglet `AlimentsCommunaute` partagé (dans `LOG_SHEET_ID`) alimenté par les ajouts clients. **Toutes les valeurs sont par GRAMME** (pas par 100g) dans les deux sources — un ajout créé côté app se fait via un formulaire "pour 100g" (plus lisible pour un client) puis divisé par 100 avant écriture (`confirmerCreationAliment`, `diete.js`). Champs : `kcal, prot, glu, sucres, fibres, lip, ags` — **la base coach (`copie de base`) n'a que kcal/prot/glu/lip**, donc `sucres/fibres/ags` valent `null` (inconnu, à ne pas confondre avec 0) pour ses entrées ; seules les entrées communautaires ont ces champs renseignés (0 par défaut si laissés vides à la création). Un ajout communautaire est visible immédiatement (utilisable tout de suite par son auteur, badge "non validé"), pas de blocage en attente de modération. Modération coach : page "Base alimentaire" (`aliments-a-valider`), `listerAlimentsCommunauteAValider`/`validerAlimentCommunaute`/`supprimerAlimentCommunaute`. Le plan écrit par le coach (`Diète` sheet) n'est jamais modifié. **Scan code-barres caméra + Open Food Facts** (`diete.js`, modale de recherche d'aliment — `ouvrirScanCodeBarre`/`onScanSuccess`) : bouton "📷 Scanner un code-barres" dans l'écran de recherche, lib `html5-qrcode` chargée en lazy depuis un CDN (iOS Safari n'a pas de `BarcodeDetector` natif). Au scan, cherche d'abord le code-barres dans la base déjà chargée (`_tousLesAliments()`, champ `codeBarre` — 12e colonne de `AlimentsCommunaute`, absente pour les entrées de la base coach) ; si absent, interroge directement `world.openfoodfacts.org` en `fetch()` (CORS ouvert, pas de proxy GAS) et préremplit le formulaire de création existant (nom + macros pour 100g) que le client confirme/ajuste avant sauvegarde via `ajouterAlimentCommunaute` (10e param `codeBarre`, optionnel) — pas d'import en masse de la base OFF, la base communautaire grossit organiquement à l'usage. `_arreterScan()` coupe explicitement la caméra à chaque sortie de l'écran scan (piège classique de ces libs : flux vidéo qui reste actif en arrière-plan sinon). **Piège** : `sheet.deleteRow(ligne)` décale les numéros de ligne de tout ce qui suit — après une suppression, toujours refetch la liste depuis le serveur plutôt que filtrer localement, sinon les `ligne` mémorisées côté client pour les entrées restantes deviennent fausses (vu aussi sur `supprimerMenu`/`supprimerSlotJournal` ci-dessous). **Piège date** : la colonne `Date` (10e) peut être auto-convertie en objet Date par Sheets même si écrite comme texte `dd/MM/yyyy HH:mm` — `listerAlimentsCommunauteAValider()` la relit via `_fmtDateHeureCommunaute()` (même motif que `_fmtDateJournal()`), sinon un `Date.toString()` JS brut ("Mon Jul 20 2026 11:03:00 GMT+0200…") remonte côté client (repéré dans la console coach).
- **Mes menus + Mon journal** (`diete.js`, page Diète à onglets Diètes/Mes menus/Mon journal) : remplace l'ancien système "ajouts libres" (1 aliment isolé dans la vue d'une diète, retiré). Le client construit des **menus** perso (plusieurs aliments de la base coach+communauté, via la même modale de recherche/création que la base alimentaire) stockés dans l'onglet `MesMenus` de son fichier — une ligne par aliment, groupée par `MenuId` (`'M'+Date.now()`, stable même si le nom change, c'est lui qui sert de référence ailleurs). Le **journal** (`JournalDiete`, même fichier) est un carnet perso jour par jour, jusqu'à 7 "repas" (slots, numéro **choisi par le client** — pas auto-assigné par ordre d'ajout, pour rester aligné sur le repas correspondant de la diète cible même si rempli dans le désordre ; `ajouterSlotJournal` refuse un slot déjà pris, `erreur:'slot_deja_rempli'`) par date choisie librement — pour chaque slot il pioche soit un Repas de la diète écrite par le coach (`Ref` = `"ligneTitre|colTitre|repasIdx"`, résolu à la volée via `chargerDieteParPosition`, jamais figé — reflète les éventuelles modifs du coach), soit un de ses menus (`Ref` = `MenuId`). **Piège rencontré** : une date `dd/MM/yyyy` écrite dans une cellule Sheets est auto-convertie en objet Date par Sheets — la relire avec `(cell+'')` donne le `Date.toString()` JS ("Mon Jul 20 2026…") au lieu de "20/07/2026", cassant le regroupement par jour côté client. `_fmtDateJournal()` (Code.js) reformate systématiquement via `Utilities.formatDate` avant de renvoyer/comparer une date de `JournalDiete` — même classe de piège que `parseDateBilan`/`listerAjoutsLibres` ailleurs dans ce fichier, à appliquer à toute nouvelle colonne de type date. Un 3e choix "🥗 Composer avec des aliments" (en plus de "Repas du coach"/"Un de mes menus") permet de construire un repas à la volée pour un slot précis, aliment par aliment, sans passer par la bibliothèque — techniquement ça appelle `creerMenu()` en coulisses avec un nom auto-généré ("Repas N du {date}" si laissé vide) puis `ajouterSlotJournal(type:'menu', ...)`, donc ça finit quand même dans `MesMenus`/la bibliothèque (pas de 3e type de stockage séparé). Un slot déjà rempli de type `'menu'` (pas `'coach'` — un repas du coach n'est jamais modifiable) est tapable pour le rouvrir en édition (`ouvrirModificationSlotJournal`, réutilise `renderJournalCompose` avec `_dMenuDraft.menuIdEdition`) — `confirmerComposeJournal` bascule alors sur `modifierMenu()` (même MenuId conservé) et, si le nom a changé, supprime/réinsère juste la ligne du slot pour rafraîchir son `Label` affiché (safe : le slot est identifié par son numéro explicite, pas par son ordre d'insertion).
- **Diète cible du journal** : chaque jour du journal peut avoir une diète cible (au plus une, `definirDieteCibleJournal`) — stockée comme une ligne "pseudo-slot" dans la même feuille `JournalDiete` (`Slot=0, Type='cible', Ref="ligneTitre|colTitre"`), pas une feuille séparée, pour rester dans le même pattern tidy-row. Une fois une cible choisie, "Mon journal" affiche des emplacements fixes "Repas 1"…"Repas N" (N = nombre de repas de la diète cible, plafonné à 7) au lieu du bouton générique "+ Ajouter un repas" — chaque emplacement affiche la cible de macros du repas correspondant de la diète cible (`cible.repas[i-1]`), pour faire des équivalences repas par repas. Sans cible sélectionnée, 7 emplacements génériques sans cible affichée (comportement par défaut). La ligne "cible" est systématiquement exclue du calcul `_joursJournal()` (compteur "N repas" du listing) et des totaux réels du jour.
- **Cible affichée pendant la construction d'un menu/repas** (`_dMenuDraft.cible`, `diete.js`) : **lecture seule, jamais un champ de saisie** (piège déjà corrigé une fois — une v1 avait des inputs éditables, retirés à la demande du user : la cible doit refléter uniquement le vrai repas de la diète sélectionnée, pas une valeur inventée à la main). Sert de repère pendant qu'on ajoute des aliments (`Cible du repas : 285/500 kcal · …`), jamais envoyée au serveur. Uniquement renseignée quand elle vient réellement du repas correspondant de la diète cible du jour (`ouvrirComposeJournal`/`ouvrirModificationSlotJournal` lisent `_cibleJournalActuelle()`) ; sans diète cible pour ce jour, ou depuis "Mes menus" (pas de lien avec un jour du journal), `_dMenuDraft.cible` reste à `null` sur tous les champs et la section ne s'affiche simplement pas.
- **Modifier un menu existant** (`modifierMenu`, Code.js) : supprime puis réinsère toutes les lignes du `MenuId` **en conservant le même MenuId** — indispensable si un slot du journal référence déjà ce menu, sinon la référence casserait. Bouton "✎" sur chaque carte de "Mes menus" (`ouvrirModificationMenu`) réutilise le même écran que la création, juste pré-rempli et avec `_dMenuDraft.menuIdEdition` renseigné. Taper sur une ligne d'aliment déjà ajoutée (au brouillon, en création comme en édition) ouvre la même modale de quantité pré-remplie (`ouvrirModifQuantiteDraft`) — les valeurs par gramme sont reconstituées en divisant les totaux stockés par la quantité, le brouillon ne garde pas le prix au gramme d'origine séparément.
- **Ne jamais confondre "en cours de chargement"/"échec réseau" avec "vraiment vide"** (`switchDieteTab`, `renderDieteErreurChargement`, `diete.js`) : `_dMenus`/`_dJournal` valent `null` tant que jamais chargés, `[]` une fois confirmés vides par le serveur. Un premier essai catchait les échecs réseau en les remplaçant silencieusement par `[]` — indiscernable d'un "vraiment vide", ce qui affichait "Aucun jour enregistré"/"Aucun menu créé" alors que les données existaient bien côté serveur, au risque de pousser le client à recréer une entrée en double. Depuis, un échec laisse `_dMenus`/`_dJournal` à `null`, et `renderDieteMenus()`/`renderDieteJournal()` affichent un écran "Erreur de chargement" avec un bouton Réessayer plutôt que la liste vide dans ce cas précis. Les deux fetch (`listerMenus`/`listerJournal`) sont lancés en parallèle (`Promise.all`) plutôt que l'un après l'autre, pour réduire le temps où l'écran de chargement plein-page reste affiché.
- **Indicateur de chargement anti double-tap** (`_guardAction(fn, btn)`, `diete.js`) : grise le bouton/l'élément cliqué et affiche un petit spinner pendant l'appel réseau, en plus de bloquer les clics concurrents (le double-tap sans feedback visuel poussait les gens à re-taper, croyant que ça n'avait pas marché). `btn` = `this` passé depuis l'attribut `onclick` — ne restaure l'élément que s'il est encore dans le DOM à la fin (le cas normal est que `setPage()` ait déjà tout remplacé). Tout nouveau bouton d'action sur cette page doit passer par `_guardAction(fn, this)`, pas `fn()` directement.
- **Garde-fou anti double-tap** (`diete.js`, `_guardAction()`) : tous les boutons d'action de la page Diète (créer/ajouter/supprimer un aliment, un menu, un slot de journal) passent par `_guardAction(fn)` au lieu d'appeler la fonction directement — un double-tap rapide sur "Créer et ajouter" créait littéralement 2 lignes identiques côté serveur (bug réel constaté et reproduit : `ajouterAlimentCommunaute`/`creerMenu`/etc. n'ont aucune protection contre les appels concurrents, chaque clic déclenche un appel réseau indépendant). Si un nouveau bouton d'action est ajouté sur cette page, l'y faire passer aussi.

## Workflow de travail

- Sync + push sans demander confirmation à chaque fois — seulement si une action est destructive (force-push, suppression) faut-il confirmer.
- Avant tout changement visuel/fonctionnel supposé "comme le GAS", relire le code GAS correspondant plutôt que deviner — plusieurs écarts trouvés cette session venaient d'un portage approximatif (ex: badges Platine/Diamant/Légendaire qui avaient perdu leurs facettes, page Classement avec des noms de champs invalides).
